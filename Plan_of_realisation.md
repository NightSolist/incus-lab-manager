# Gap-анализ: текущая реализация vs. требования главы 2 ВКР

## 1. Функциональные требования

| № | Требование | Статус | Что есть | Что отсутствует |
|---|-----------|--------|----------|-----------------|
| ФТ-1 | Мониторинг изменений в репозитории Incus (`shared/api/` и `client/`) | ❌ Не реализовано | — | Модуль `monitor.py` полностью отсутствует |
| ФТ-2 | Парсинг Go-структур | ✅ Реализовано частично | `ASTParser` + Go-инструмент `go-ast-parser`; `RegexParser` как запасной | Нет загрузки файлов через GitHub API; работает только с локальным репозиторием; нет разрешения cross-file зависимостей через API |
| ФТ-3 | Шаблонная генерация (Jinja2) | ⚠️ Частично | `RustGenerator` генерирует структуры, enum, alias | Нет Jinja2-шаблонов; генерация реализована через конкатенацию строк, а не шаблоны; нет генерации комментариев с метаданными (SHA, дата, метод) |
| ФТ-4 | LLM-генерация через Ollama | ❌ Не реализовано | — | Модуль `llm_generator.py` полностью отсутствует |
| ФТ-5 | Автоматический выбор метода генерации (диспетчер) | ❌ Не реализовано | Есть whitelist `TARGET_ENTITIES` в `pipeline.py` | Модуль `dispatcher.py` отсутствует; нет проверки доступности Ollama; нет классификации структур по сложности |
| ФТ-6 | Валидация сгенерированного кода | ⚠️ Частично | `RustValidator` запускает `cargo check` на всём наборе файлов | Нет синтаксической проверки (уровень 1); нет проверки полноты полей (уровень 2); нет извлечения текста ошибок для LLM; нет цикла повторной генерации; нет разделения результатов по структурам |
| ФТ-7 | Создание Pull Request | ❌ Не реализовано | — | Модуль `publisher.py` полностью отсутствует |
| ФТ-8 | Отчётность об изменениях `client/` | ❌ Не реализовано | — | Зависит от ФТ-1 и ФТ-7 |
| ФТ-9 | Уведомление о структурах, требующих ручной обработки | ❌ Не реализовано | — | Зависит от ФТ-6 и ФТ-7 |
| ФТ-10 | Email-уведомления через Woodpecker CI | ❌ Не реализовано | — | Нет `.woodpecker.yml`; нет шага `notify-email` |
| ФТ-11 | Ведение журнала выполнения | ⚠️ Частично | Есть `print`-вывод в `pipeline.py` | Нет структурированного журнала; нет временных меток; нет сохранения в файл |

## 2. Архитектурные расхождения

| Компонент | В тексте ВКР | В реальной реализации | Критичность |
|-----------|-------------|----------------------|-------------|
| Инструмент парсинга | Python `tree-sitter` | Go `go/ast` + subprocess | 🔴 Высокая — прямо противоречит п. 2.4.1 |
| Источник данных | GitHub API (PyGithub) | Локальная копия репозитория | 🔴 Высокая — противоречит ФТ-1, ФТ-2 |
| Шаблонизация | Jinja2 с файлами `.j2` | Конкатенация строк в Python | 🟡 Средняя — функционально эквивалентно, но не соответствует описанию |
| Диспетчер | Отдельный модуль `dispatcher.py` | Whitelist в `pipeline.py` | 🟡 Средняя — логика есть, но не выделена |
| LLM-генератор | Ollama + Qwen2.5-Coder | Отсутствует | 🔴 Высокая — ключевой компонент системы |
| Валидатор | Многоуровневый с циклом регенерации | Однопроходный `cargo check` | 🟡 Средняя — базовая функциональность есть |
| CI/CD | Woodpecker CI + `.woodpecker.yml` | Отсутствует | 🔴 Высокая — ключевой инфраструктурный компонент |
| GitHub интеграция | PyGithub (мониторинг + PR) | Отсутствует | 🔴 Высокая — ключевой компонент системы |
| Email-уведомления | `drillster/drone-email` в CI | Отсутствует | 🟡 Средняя — важный, но не блокирующий компонент |
| Структура Rust-проекта | `src/incus/generated/` + `src/incus/custom/` | Плоская структура `src/incus/` | 🟡 Средняя — требует рефакторинга |
| Конфигурация | `config.yaml` с полным набором параметров | Минимальный `Config` dataclass | 🟡 Средняя — легко доработать |
| Файл состояния | `.sync_state` в репозитории | Отсутствует | 🔴 Высокая — без него мониторинг невозможен |

## 3. Нефункциональные требования

| № | Требование | Статус | Комментарий |
|---|-----------|--------|-------------|
| НФТ-1 | Только открытые инструменты | ✅ | Все используемые инструменты открытые |
| НФТ-2 | Автономность инфраструктуры | ⚠️ | Локальная работа есть, но GitHub API и CI не подключены |
| НФТ-3 | Время выполнения ≤ 30 минут | ⚠️ | Локальный pipeline быстрый, но с LLM и GitHub API не проверялось |
| НФТ-4 | Воспроизводимость шаблонного генератора | ✅ | Генератор детерминирован |
| НФТ-5 | Изоляция окружения (Docker) | ❌ | CI/CD отсутствует, Docker-образы не определены |
| НФТ-6 | Устойчивость к частичным отказам | ❌ | При ошибке генерации pipeline падает целиком |
| НФТ-7 | Расширяемость | ⚠️ | `ParserFactory` хорошо масштабируется; остальное жёстко захардкожено |
| НФТ-8 | Требования к ресурсам CI | ❌ | CI отсутствует |

## 4. Ограничения

| № | Ограничение | Соответствие | Комментарий |
|---|------------|-------------|-------------|
| О-1 | Только модели данных, `client/` только мониторится | ✅ | Клиентская логика не генерируется |
| О-2 | Зависимость от `gofmt`-форматирования | ✅ | AST-парсер работает корректно с форматированным кодом |
| О-3 | Только GitHub API и SMTP как внешние сервисы | ⚠️ | Пока вообще нет внешних сервисов |
| О-4 | Go 1.21+ | ✅ | `go/ast` поддерживает все актуальные версии |
| О-5 | Доступность Ollama | ❌ | Ollama не интегрирована |
| О-6 | Лимит попыток регенерации ≤ 3 | ❌ | Регенерации нет |
| О-7 | Только компиляционная валидация | ✅ | Именно это и реализовано |
| О-8 | Однопоточная обработка | ✅ | Pipeline работает последовательно |

## 6. Что можно оставить как ограничение прототипа

Следующие вещи **допустимо не реализовывать полностью** — достаточно упомянуть в тексте как ограничение:

- полноценный LLM-цикл с тремя попытками регенерации (достаточно одной попытки);
- обработку rate limit GitHub API;
- force-push и конфликты при коммите;
- параллельную обработку структур;
- функциональное тестирование сериализации.
